# 「一句话钩子」(Why-Watch Hook) — Design

**Date:** 2026-06-17
**Status:** Approved, ready for implementation plan
**Roadmap position:** Idea **A** of the A→B→C evolution (A 一句话钩子 → B 灵感碰撞 → C 今日速读). A is the foundation; B and C build on it.

## Problem

Atlas is strong at *collecting* and *querying* AI content, but the browse
experience is thin: each item in a list is just a title + author. Video cards
show **no description at all**; blog rows show an objective `summary`. Scanning a
list gives little sense of *why any given item is worth your time*.

The guiding principle for this evolution: **简洁 + 实用 + 好玩，拒绝花哨** — distill,
don't decorate. So the fix is not a new subsystem but a small, high-leverage
addition that makes every item legible at a glance, reusing infrastructure that
already exists.

## Goal

Every browse item shows a one-line, punchy **"为什么值得看"** hook beneath its
title — an editorial *angle/reason*, distinct from the descriptive `summary`.
Generated by the same LLM pipeline that already produces graph-node aliases, and
a **silent no-op** when `ATLAS_CHAT_*` is unconfigured.

## Key architectural fact

Browse/home pages render from **contract files loaded per request**
(`load_items`), while the hook (like `alias`) lives in the **derived SQLite DB**.
So the feature needs a tiny bridge: read a `{id: hook}` map from the DB by id and
inject it into the template context. Files remain the single source of truth; the
hook is derived and rebuildable (`python -m aishelf.db sync --rebuild`).

This mirrors the existing `alias` rollout exactly — the design is deliberately
"the alias feature again, but one sentence instead of ≤8 chars, shown in browse
lists instead of graph nodes."

## Design decisions (locked)

- **Display placement:** blog rows show the hook **above** the existing
  `summary` (hook = angle/reason, summary = objective description; they
  complement). Video cards (which have no description today) gain the hook line.
- **Regeneration policy:** mirror `alias` — regenerate when the item is new, its
  **title changed**, or it has no hook. A **note-only edit does NOT** regenerate
  the hook (avoids spending LLM budget on note edits). Accepting that a
  summary-only change won't refresh the hook is a deliberate simplicity tradeoff
  — rare for collected content.
- **Separate LLM call from alias:** `generate_hooks` is its own batched call in
  the same `alias.py` module — independent failure, clearer code. Both only run
  on new/changed items, batched, so cost is bounded.
- **Zero new config:** reuses `alias.is_configured()` (`ATLAS_CHAT_*`). No new
  env var.
- **Length:** hook target ≤30 Chinese chars; a safety cap of `HOOK_MAX_CHARS`
  (40) trims overruns (mirrors `ALIAS_MAX_CHARS`).

## Components

### 1. LLM generation — `src/aishelf/alias.py`

Add `generate_hooks(pairs: list[tuple[str, str]]) -> list[str] | None`, a near
twin of `generate_aliases`:

- Same gate (`get_alias_settings()` / `is_configured()`), same client, same
  batching (`ALIAS_BATCH_SIZE`), same all-or-nothing per-batch contract (each
  batch must return a JSON array of equal length, else `None`).
- Returns `None` when unconfigured, `pairs` empty, on any error, or on
  parse/length mismatch — caller then simply renders no hook.
- New prompt (distinct from the alias prompt), roughly:
  > 为下列每条内容各写一句话，说明它为什么值得一看（不超过30个汉字），要具体、有信息量、能勾起兴趣，不要空话套话。只返回一个 JSON 字符串数组，元素顺序与输入编号一致，不要输出任何额外文字或解释。
- Reuse `_extract_json_array`; add a `_clean`-style trim capped at
  `HOOK_MAX_CHARS = 40`.

### 2. DB schema — `src/aishelf/db/schema.py`

Add `hook TEXT` (nullable) to the `items` table, alongside `alias`. Backfilled by
`sync --rebuild`. No FTS change (hooks are not searched).

### 3. Sync — `src/aishelf/db/sync.py`

Mirror the alias path within the existing single transaction:

- Extend the `existing` fetch with
  `(hook IS NOT NULL AND hook != '') AS has_hook`.
- `needs_hook = alias.is_configured() and (prev is None or it.title != prev_title or not has_hook)`.
- Include `needs_hook` in the "skip if nothing to do" short-circuit.
- Collect `to_hook` items; after the alias block, call
  `alias.generate_hooks([(it.title, it.summary) for it in to_hook])` and
  `UPDATE items SET hook=? WHERE id=?` for each, in the same transaction.
- On `None`/length-mismatch: log a warning and leave rows intact (never crash
  sync) — identical to the alias failure handling.

### 4. DB read — `src/aishelf/db/search.py`

Add `hooks_for(ids: list[str], db_path=None) -> dict[str, str]`:

- One query: `SELECT id, hook FROM items WHERE id IN (…) AND hook IS NOT NULL AND hook != ''`.
- Read-only; returns `{}` on empty input or a missing/unopenable DB (tolerant,
  like the rest of the derived-view reads).

### 5. Site wiring — `src/aishelf/site/app.py`

- Add a small helper `_hooks_for(items)` next to `_note_for`, wrapping
  `db_search.hooks_for([it.id for it in items])` in a `try/except` so a
  missing/locked DB never breaks browsing.
- Pass `hooks=_hooks_for(...)` into the template context for the routes that
  render file-loaded lists via the shared partials: `/` (home), `/videos`,
  `/blogs`, `/authors/{key}`.
- **Out of scope for v1:** `/search` (renders via `search.html`, different path)
  and `/ask`. Easy to add later.

### 6. Templates + styles

- `_video_card.html`: add, beneath the `sub` line,
  `{% if hooks and hooks.get(it.id) %}<span class="hook">{{ hooks[it.id] }}</span>{% endif %}`.
- `_blog_row.html`: add the same hook line **above** `<p class="summary">`.
- `style.css`: a `.hook` rule — subtle accent (e.g. a left vertical bar `▍` /
  `border-left`, muted-accent color, smaller weight). Guard all template access
  with `hooks and hooks.get(...)` so pages that don't pass `hooks` render
  unchanged. `{% include %}` inherits parent context, so home/list/author pages
  work once their route passes `hooks`.

## Data flow

```
Hermes / manual write → data/{videos,blogs}/<id>.json   (source of truth)
        → db sync: generate hook into items.hook (only when ATLAS_CHAT_* set)
        → browse route: hooks_for(ids) → {id: hook}
        → template renders the hook line
```

## Degradation

`ATLAS_CHAT_*` unset → no hooks generated → `hooks` map empty → every page
renders exactly as it does today. Zero new config, zero new crash surface.

## Testing

(Network/LLM always mocked, per repo convention.)

- **`generate_hooks`** — five cases mirroring the alias tests: configured happy
  path (mock client returns a matching JSON array); unconfigured → `None`; empty
  `pairs` → `None`; length mismatch → `None`; client error → `None`.
- **`sync`** (mock `alias.generate_hooks`): hook generated for a new item;
  regenerated when the title changes; **not** regenerated on a note-only edit;
  rows left intact when generation returns `None`; no hooks when unconfigured.
- **`hooks_for`**: returns the id→hook map; skips null/empty hooks; returns `{}`
  for a missing DB / empty input.
- **Route/template**: a browse page passes `hooks` and the rendered HTML shows
  the hook line (TestClient against a seeded DB, or by mocking `_hooks_for`).

## Non-goals (YAGNI)

- No hook on detail pages (you're already there).
- No regeneration on summary-only change (title-keyed, mirrors alias).
- Not shown on `/search` results or `/ask` in v1.
- No hand-editing of hooks.
- No new environment variable.

## Migration

After deploy, run once: `python -m aishelf.db sync --rebuild` to backfill the
`hook` column (same step the alias rollout required).
