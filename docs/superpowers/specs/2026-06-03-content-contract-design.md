# Content Data Contract — Design Spec

**Date:** 2026-06-03
**Status:** Approved (pending spec review)

## Background: project re-plan

aishelf is being re-planned. The original three-module plan (custom Python
scraper → LLM understanding → web frontend) is replaced by:

1. **Collection** — delegated to the **Hermes agent**, which crawls sources and
   writes content records to the filesystem. No custom scraper code.
2. **Display site** — a general content site (video-site + blog-site hybrid)
   that reads the collected records and presents them.

Two content modalities: **videos** scraped from sites (YouTube, Bilibili, …) and
**blog/document** articles.

The two halves are decoupled by a **data contract**: Hermes writes records in a
defined layout + schema; the display site reads them. This spec defines that
contract. It is the foundation sub-project; the display site and the Hermes
collection orchestration are separate specs built on top of it.

## Scope

In scope:

- On-disk storage layout for collected content.
- The field schema for the two modalities (video, blog).
- A machine-readable contract in code: pydantic models + a loader that reads and
  validates records.
- Example fixtures so the display site can be developed without Hermes.
- An exported JSON Schema document to hand to Hermes as its output target.

Out of scope (other sub-projects / YAGNI):

- The display site itself (rendering, search UI, pagination).
- How Hermes is triggered/orchestrated.
- Downloading video files or full article bodies — by decision, only metadata
  and a link to the source are stored (videos), and a summary + link (blogs).
- A database. Storage is flat JSON files; any index is built by the display
  side at read time.

## Decisions (locked during brainstorming)

- **Video modality:** store metadata + source link only (no file download).
  Display links out / embeds the source player.
- **Blog modality:** store a summary + link only (no full article body).
- **Storage:** flat JSON files on the filesystem, one record per file, grouped
  only by modality. All browse/filter dimensions live in fields, not in the
  directory structure.
- **Browse dimensions that matter:** author, published time, title, summary,
  keywords. `summary` and `keywords` are generated by Hermes during collection
  (the source does not reliably provide them).

## Storage Layout

```
data/
  videos/<id>.json       # one video record per file
  blogs/<id>.json        # one blog record per file
  _failures/<id>.json    # optional: records Hermes could not fully collect
```

- **`<id>` = `<platform>-<native_id>`** — globally unique and stable. Examples:
  `youtube-dQw4w9WgXcQ`, `bilibili-BV1xx411c7mD`. For blogs with no native id,
  use `blog-<sha1(source_url)[:12]>`. The filename stem equals the `id` field.
- **Idempotent:** re-collecting the same item overwrites the same file.
- **Atomic write (Hermes side):** write `<id>.json.tmp`, then `os.rename` to
  `<id>.json`. The presence of the final `<id>.json` is the success signal, so a
  reader never sees a half-written file. (Mirrors the original M1 atomic-write
  convention.)
- **`_failures/`** records carry at least `id`, `source_url`, and an `error`
  string. They are not loaded as content; they exist for debugging collection.

## Field Schema

### Shared core fields (required on every record)

| Field | Type | Notes |
|---|---|---|
| `id` | string | Unique stable id; equals the filename stem |
| `type` | `"video"` \| `"blog"` | Modality discriminator |
| `title` | string | |
| `author` | string | Author / account display name |
| `platform` | string | `youtube` / `bilibili` / blog site domain |
| `source_url` | string (URL) | Canonical link to the original |
| `published_at` | string | Original publish date, ISO 8601 (`YYYY-MM-DD` or full datetime) |
| `summary` | string | Hermes-generated summary |
| `keywords` | string[] | Hermes-generated keywords (may be empty list) |
| `collected_at` | string | Collection timestamp, ISO 8601 datetime |

### Video-only fields

| Field | Type | Required | Notes |
|---|---|---|---|
| `thumbnail_url` | string (URL) | yes | Remote thumbnail image |
| `duration_seconds` | integer | no | |
| `embed_url` | string (URL) | no | If present, display can embed a player; else it links to `source_url` |
| `author_id` | string | no | Stable account id for grouping by author |

### Blog-only fields

| Field | Type | Required | Notes |
|---|---|---|---|
| `cover_image_url` | string (URL) | no | Card cover image |
| `site_name` | string | no | Human-readable site name |
| `author_id` | string | no | Stable account id for grouping by author |

Validation rules:

- `published_at` accepts either a date (`YYYY-MM-DD`) or an ISO 8601 datetime;
  stored as a string, parsed leniently by the loader.
- `keywords` is always a list (possibly empty), never a comma-joined string.
- Unknown extra fields are ignored (forward-compatible), not rejected.

## Code Contract

The contract is also expressed in code so both sides stay aligned.

### `src/aishelf/contract/models.py`

Pydantic v2 models:

- `ContentItem` — base with the shared core fields.
- `VideoItem(ContentItem)` — adds video-only fields; `type` fixed to `"video"`.
- `BlogItem(ContentItem)` — adds blog-only fields; `type` fixed to `"blog"`.
- A discriminated union on `type` for parsing an arbitrary record into the right
  subtype.

Models ignore unknown fields and validate required fields / types. Invalid input
raises a pydantic `ValidationError`.

### `src/aishelf/contract/loader.py`

- `load_items(data_dir) -> list[ContentItem]` — reads `data/videos/*.json` and
  `data/blogs/*.json`, validates each against the matching model, returns the
  typed records sorted by `published_at` descending.
- **Robustness:** a file that is malformed JSON or fails validation is skipped
  and logged (with its path and the error); it never aborts the whole load. This
  keeps a single bad record from breaking the display site.
- `_failures/` is not read by `load_items`.

### Fixtures

A small set of realistic sample records committed under a fixtures location
(e.g. `tests/fixtures/contract/videos/*.json` and `.../blogs/*.json`) — at least
two valid videos, two valid blogs, and one malformed file — used by loader tests
and available for the display site to develop against.

### Exported JSON Schema

A committed JSON Schema document (e.g.
`docs/contract/content-item.schema.json`), generated from the pydantic models,
to hand to Hermes as the authoritative "produce records shaped like this" spec.

## Error Handling

- **Reader (loader):** malformed JSON or validation failure → skip + log, continue.
- **Writer (Hermes):** atomic write prevents partial reads; items that cannot be
  fully collected are written to `_failures/` with an `error` field rather than
  emitting an incomplete content record.

## Testing

- **Models:** valid payloads parse into the correct subtype; missing required
  fields and wrong types raise `ValidationError`; unknown extra fields are
  ignored; the `type` discriminator routes to `VideoItem` / `BlogItem`.
- **Loader:** given the fixtures dir, returns all valid items typed correctly,
  skips the malformed file, and sorts by `published_at` descending. A bad file
  does not raise.
- **JSON Schema export:** the generated schema file is up to date with the models
  (a test regenerates and compares).

No Hermes / network involvement in any test.

## Success Criteria

- The pydantic models and loader exist and are importable.
- `load_items` reads the fixtures, returns typed records, and tolerates a bad file.
- A JSON Schema document is exported and committed.
- The display-site sub-project can be built and tested entirely against fixtures
  conforming to this contract, with no dependency on Hermes.
